- Resolved: fix tokenizer dependency loading. Token counting now loads runtime deps from normal installed `node_modules` instead of expecting a non-existent `vendor/node_modules` tree.
- Resolved: fix the npm installation story. The published npm package can now use its installed runtime dependencies directly, so npm installs no longer fall back to approximate counting just because `vendor/` is missing.
- Resolved: fix session/subagent telemetry aggregation. OpenCode stores per-API-call `step-finish` parts, while `assistant.info.tokens` only reflects the latest step on that assistant message. TokenScope now sums the per-call `step-finish` data instead of undercounting multi-step turns.
- Remaining upstream limitation: OpenCode local session/export payloads only persist aggregate `tokens.cache.write` per API call. The Anthropic `cache_write_5m` / `cache_write_1h` split exists in console billing/provider code, but it is not stored in local session messages, so TokenScope cannot recover that breakdown post hoc without upstream storage changes.
- Remaining upstream limitation: OpenCode also does not persist finish-step provider metadata in exported session history. Some providers expose extra response-side numbers (for example accepted/rejected prediction tokens or service-tier metadata), but those values are dropped before session export, so they are not available for after-the-fact TokenScope reporting today.
